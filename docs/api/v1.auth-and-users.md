# Auth и учётные записи (v1) — контракт для фронтенда

Базовый префикс: **`/api/v1`**.  
Формат тел и ответов — **JSON**, имена полей в **camelCase** (как в Swagger и в типичном ответе ASP.NET Core).

Источник истины по полям DTO — **Swagger** (`/swagger`) у запущенного API.

### CORS (локальная разработка)

Разрешённые origins задаются в конфигурации **`Cors:AllowedOrigins`** (массив строк). В **`appsettings.Development.json`** по умолчанию указан типичный origin Angular: **`http://localhost:4200`**. Для продакшена добавьте реальный URL фронта в конфигурацию среды.

## 1. Вход и сессия

### POST `/api/v1/auth/login`

- **Аутентификация**: не требуется.
- **Тело**:

```json
{
  "email": "user@example.com",
  "password": "YourPassword1"
}
```

- **Успех 200** — объект сессии (те же поля, что в `AuthResponse`):

| Поле (JSON) | Тип | Назначение |
|-------------|-----|------------|
| `accessToken` | string | JWT; передавать в заголовке **`Authorization: Bearer <accessToken>`** |
| `refreshToken` | string | Одноразовый маркер сессии в Redis; хранить на клиенте безопасно |
| `userId` | string (UUID) | Идентификатор пользователя |
| `email` | string | Email (нормализованный на сервере) |
| `role` | string | Системное имя роли (`Student`, `Teacher`, `DepartmentHead`, `Admin`, …) |

- **401** — неверные учётные данные или деактивированный пользователь (`ProblemDetails`).
- **400** — пустой email/пароль и т.п. (модель не прошла валидацию).
- **429** — слишком частые попытки входа с одного IP (rate limiting). Клиенту: backoff / сообщение пользователю.

### POST `/api/v1/auth/refresh`

- **Аутентификация**: не требуется.
- **Тело** (имя поля зафиксировано атрибутом):

```json
{
  "refreshToken": "<значение из прошлого ответа login или refresh>"
}
```

- **Успех 200** — новая пара **`accessToken`** + **`refreshToken`**; старый `refreshToken` после успешного ответа **недействителен** (ротация).
- **401** — неверный/истёкший refresh, пользователь удалён или деактивирован.

**Рекомендуемый поток на клиенте**

1. После `login` сохранить `refreshToken` (и при желании время жизни access из JWT или фиксированный интервал обновления до истечения).
2. При **401** на любом защищённом запросе с access-токеном — один раз вызвать `refresh` с сохранённым `refreshToken`, подставить новый `accessToken` и повторить запрос; если `refresh` вернул 401 — считать сессию завершённой и вести на экран входа.

### POST `/api/v1/auth/logout`

- **Тело**: так же `{ "refreshToken": "..." }`.
- **Успех 204** — refresh отозван; последующий `refresh` с этим значением даст **401**.

## 2. Создание пользователя (только администратор)

Публичной регистрации **нет**. Учётки создаёт администратор (или отдельный сценарий с SSO вуза — вне текущего API).

### POST `/api/v1/users`

- **Аутентификация**: **`Authorization: Bearer <accessToken>`** пользователя с ролью **`Admin`**.
- **Тело**:

```json
{
  "email": "new.user@example.com",
  "password": "CompliantPass1",
  "firstName": "Имя",
  "lastName": "Фамилия",
  "middleName": null,
  "roleId": "00000000-0000-0000-0000-000000000000"
}
```

- **`roleId`** — UUID строки из справочника ролей (список: **`GET /api/v1/user-roles`**, только **JWT администратора**). Несуществующий `roleId` → **400**.
- **Пароль** (сервер): длина **10–128**, без пробелов, минимум **одна буква** и **одна цифра**; email — валидный формат после нормализации.
- **Успех 201** — без токенов, только данные созданного пользователя:

| Поле | Тип |
|------|-----|
| `userId` | UUID |
| `email` | string |
| `role` | string (CodeName роли) |

- **409** — email уже занят.
- **401 / 403** — нет токена или не админ.

После создания пользователь входит обычным **`POST .../auth/login`**.

## 3. Защищённые запросы

Для всех маршрутов под **`[Authorize]`** (fallback по умолчанию):

```http
Authorization: Bearer <accessToken>
```

Cookie-сессия API **не используется** — только заголовок Bearer.

## 4. Health (для отладки и мониторинга)

Кратко см. [health.md](health.md).

- **`GET /health`** — без токена, лёгкий smoke.
- **`GET /health/db`** — проверка подключения к PostgreSQL: нужен **либо** JWT с ролью **Admin**, **либо** заголовок **`X-Health-Probe-Key`**, если в конфигурации API задан непустой **`Health:DbProbeKey`**. Иначе **401**.

---

## 5. Ошибки и лимиты

- Типичные коды: **400** (валидация), **401** (нет/просрочен токен, неверный refresh), **403** (нет прав), **409** (конфликт), **429** (rate limit на `login` / `refresh`).
- Тело ошибок — **ProblemDetails** (поля `title`, `detail`, `status`, при необходимости `instance`).

При **500** в ответе также приходит Problem Details; в **Development** в `detail` может быть больше технической информации.
