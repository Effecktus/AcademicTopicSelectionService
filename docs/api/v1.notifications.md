# Notifications (v1)

Inbox API для пользовательских уведомлений.

Базовый путь: `/api/v1/notifications`

## Эндпоинты

### `GET /api/v1/notifications`

Получить уведомления текущего пользователя (по JWT).

Query-параметры:
- `isRead` (`true|false`, необязательный) — фильтр по статусу прочтения;
- `page` (по умолчанию `1`);
- `pageSize` (по умолчанию `50`, ограничение `1..200`).

Ответ: `PagedResult<NotificationDto>`.

### `PUT /api/v1/notifications/{id}/read`

Отметить одно уведомление как прочитанное.

- `204 NoContent` — успех;
- `404 NotFound` — уведомление не найдено;
- `403 Forbidden` — попытка отметить чужое уведомление.

### `PUT /api/v1/notifications/read-all`

Отметить все уведомления текущего пользователя как прочитанные.

- `204 NoContent` — успех.

## Где создаются уведомления

На текущем этапе уведомления создаются в бизнес-потоках:

- `SupervisorRequests`:
  - создание запроса студентом → уведомление преподавателю;
  - approve/reject преподавателем → уведомление студенту;
  - cancel студентом → уведомление преподавателю.
- `StudentApplications`:
  - создание заявки студентом → уведомление преподавателю;
  - reject научруком → уведомление студенту;
  - **approve научруком** → уведомление заведующему кафедрой (`ApplicationSubmittedToDepartmentHead`, заявка уже в `PendingDepartmentHead`);
  - approve/reject заведующим кафедрой → уведомления студенту и научному руководителю (`ApplicationStatusChanged`, `SupervisorDecisionByDepartmentHead`).

Email-отправка выполняется асинхронно через фонового воркера и очередь задач.
