# Описание выбранных инструментальных средств разработки с обоснованием и сравнительным анализом аналогов

**Тема ВКР:** Разработка прототипа сервиса по выбору научного руководителя и темы выпускной квалификационной работы в высшем учебном заведении.

Данный материал предназначен для включения в пояснительную записку к ВКР (раздел «Разработка программного продукта» или глава 2–3 в зависимости от структуры работы).

---

## 1. Описание выбранных инструментальных средств разработки

Выбор инструментальных средств определяется платформой (веб-приложение), архитектурой (клиент-серверная, многослойная по принципу Clean Architecture), требованиями к надёжности, безопасности и сопровождаемости, а также целесообразностью внедрения в вузе. Ниже приведено описание выбранного стека с обоснованием.

### 1.1. Серверная часть (backend)

**Платформа и язык программирования.** Для реализации серверной части выбран **ASP.NET Core 10** (на базе .NET 10) и язык **C#**. ASP.NET Core — кроссплатформенный фреймворк для построения веб-приложений и REST API, поддерживаемый Microsoft и сообществом с открытым исходным кодом [1].

Обоснование выбора:
- Встроенная поддержка построения Web API, маршрутизации, внедрения зависимостей (DI) и конфигурации, что ускоряет разработку и обеспечивает единообразие архитектуры.
- Строгая типизация и современные конструкции C# снижают количество ошибок на этапе компиляции и упрощают рефакторинг.
- Высокая производительность и низкое потребление ресурсов по сравнению с рядом альтернатив, что важно для развёртывания в условиях ограниченной инфраструктуры вуза.
- Долгосрочная поддержка (LTS) и предсказуемый цикл обновлений платформы .NET.
- Хорошая интеграция с Entity Framework Core для работы с реляционной базой данных, встроенные механизмы аутентификации и авторизации.

**Доступ к данным.** Для объектно-реляционного отображения (ORM) и работы с базой данных используется **Entity Framework Core 10**. EF Core позволяет описывать модель данных в виде классов C#, генерировать и применять миграции схемы, выполнять запросы через LINQ и обеспечивать отслеживание сущностей (change tracking). В проекте схема базы данных задаётся явно (SQL-скрипты в составе инфраструктуры), а EF Core используется как ORM поверх этой схемы, что соответствует принципам контролируемой миграции данных в командной разработке.

**Документирование API.** Для описания и интерактивной документации REST API применяется **Swagger (OpenAPI)**. Это обеспечивает единый контракт интерфейса, удобную проверку эндпоинтов в процессе разработки и возможность генерации клиентского кода при необходимости.

### 1.2. Клиентская часть (frontend)

**Фреймворк и язык.** Для разработки пользовательского интерфейса выбран **Angular 18** и язык **TypeScript**. Стилизация выполняется с помощью **SCSS** [2].

Обоснование выбора:
- Angular — полнофункциональный фреймворк с встроенной маршрутизацией, формами, HTTP-клиентом и внедрением зависимостей, что обеспечивает структурированность одностраничного приложения (SPA) и предсказуемую организацию кода.
- TypeScript добавляет статическую типизацию поверх JavaScript, снижая количество ошибок на этапе разработки и упрощая рефакторинг и навигацию по проекту.
- Широкая распространённость в корпоративной и образовательной среде, обширная документация и совместимость с современными браузерами.
- Удобство поддержки крупных форм (заявки, каталог тем, чат) и разграничения интерфейсов по ролям (студент, преподаватель, заведующий кафедрой, администратор).

### 1.3. Система управления базами данных

В качестве СУБД выбрана **PostgreSQL 16** [3].

Обоснование выбора:
- Реляционная модель данных соответствует структурированному хранению пользователей, кафедр, тем ВКР, заявок, сообщений чата и метаданных архива ВКР; поддержка транзакций и целостности данных (первичные и внешние ключи, ограничения) снижает риск несогласованности.
- Высокая надёжность, производительность при сложных запросах и больших объёмах данных, возможность тонкой настройки через индексы и анализ планов выполнения.
- Открытая лицензия и отсутствие лицензионных затрат при внедрении в вузе.
- Широкая поддержка со стороны облачных провайдеров и инструментов развёртывания (Docker), хорошая интеграция с EF Core.

### 1.4. Вспомогательные инструменты и сервисы

**Контейнеризация.** Для единообразного развёртывания и изоляции сервисов используется **Docker** и **Docker Compose**. Контейнеры задают окружение для API, PostgreSQL, Redis, MinIO и мониторинга, что упрощает запуск полного стека в среде разработки и на сервере без конфликтов версий.

**Кэширование.** **Redis 7** используется для кэширования сессий, данных частых запросов и при необходимости очередей, что снижает нагрузку на БД и ускоряет отклик приложения.

**Файловое хранилище.** Для хранения файлов архива ВКР (документы, презентации) используется **MinIO** в среде разработки и **AWS S3** (или совместимый сервис) в промышленной среде. Это обеспечивает масштабируемое хранение бинарных данных вне основной БД и соответствие типовым практикам облачных приложений.

**Авторизация и безопасность.** Авторизация реализована на основе **JWT (JSON Web Token)** и **Refresh Tokens**, с разграничением доступа по ролям (Role-Based Access Control). Такой подход не требует хранения сессий на сервере, упрощает горизонтальное масштабирование и соответствует распространённой практике для REST API.

**Уведомления.** Для оповещения пользователей о смене статусов заявок и новых сообщениях в чате используется отправка электронной почты через **MailKit (SMTP)**.

**Контроль версий и сборка.** Исходный код хранится в **Git**; сборка backend выполняется средствами **dotnet CLI**, frontend — с использованием **Angular CLI**. Приветствуется использование средств автоматизированного тестирования (например, xUnit/NUnit на backend, Jasmine/Karma или Jest на frontend) и непрерывной интеграции (CI) по рекомендациям методических указаний [4].

### 1.5. Среда разработки

Разработка серверной части ведётся в **JetBrains Rider**, который обеспечивает полноценную поддержку отладки, рефакторинга и работы с EF Core для проектов на C# и .NET. Клиентская часть (Angular, TypeScript) разрабатывается в **Visual Studio Code** с расширениями для Angular и TypeScript.

## 2. Сравнительный анализ аналогов

Ниже рассмотрены основные аналоги выбранных инструментальных средств по категориям: серверная платформа, клиентский фреймворк и СУБД. Сравнение проведено по критериям, значимым для проекта прототипа сервиса по выбору научного руководителя и темы ВКР в вузе.

### 2.1. Обзор аналогов по категориям

#### 2.1.1. Серверные платформы (backend)

**Java (Spring Boot).** Платформа Java с фреймворком Spring Boot широко применяется для корпоративных веб-приложений и REST API [5]. Достоинства: богатая экосистема, строгая типизация, высокая производительность JVM, развитая поддержка безопасности и работы с БД (Spring Data, Hibernate). Недостатки для данного проекта: более высокая потребность в оперативной памяти, более сложная начальная настройка по сравнению с ASP.NET Core, необходимость выбора множества подмодулей Spring. Для прототипа с чётко очерченным объёмом функций и возможным развёртыванием на ограниченных ресурсах вуза ASP.NET Core обеспечивает быстрый старт и достаточную функциональность без избыточности.

**Node.js (Express, NestJS).** Платформа на базе JavaScript/TypeScript с асинхронной моделью выполнения [6]. Достоинства: один язык на backend и frontend, высокая скорость разработки для I/O-нагруженных сценариев, большое количество пакетов в экосистеме npm. Недостатки: слабая типизация в чистом JavaScript (частично компенсируется TypeScript), менее строгая архитектура по умолчанию, необходимость тщательного выбора библиотек для безопасности и работы с БД. Для приложения с выраженной бизнес-логикой, транзакциями и ролевой моделью предпочтительна платформа с жёсткой типизацией и встроенной поддержкой этих аспектов, что обеспечивает выбранный стек .NET.

#### 2.1.2. Клиентские фреймворки (frontend)

**React.** Библиотека для построения пользовательских интерфейсов с компонентным подходом и виртуальным DOM [7]. Достоинства: огромное сообщество, гибкость в выборе маршрутизации, управления состоянием и HTTP-клиента. Недостатки для структурированного учебного/корпоративного проекта: необходимость самостоятельного выбора и согласования стека (роутинг, формы, состояние), что увеличивает вариативность и риск несогласованности решений. Angular предоставляет единый полнофункциональный фреймворк с согласованными практиками, что упрощает поддержку и соответствие стандартам оформления ВКР.

**Vue.js.** Прогрессивный фреймворк для построения SPA с мягкой кривой обучения [8]. Достоинства: простота внедрения, хорошая документация, удобство для небольших и средних проектов. Недостатки в контексте данной работы: меньшая по сравнению с Angular стандартизация структуры крупного приложения (несколько ролей, множество экранов, формы заявок, чат), менее выраженная ориентация на корпоративную разработку и типизацию. Для прототипа с чётким набором ролей и сценариев Angular даёт более предсказуемую архитектуру и удобство документирования в пояснительной записке.

#### 2.1.3. Системы управления базами данных

**MySQL / MariaDB.** Популярные реляционные СУБД с открытой лицензией [9]. Достоинства: широкое распространение, простота развёртывания, хорошая производительность для типовых веб-нагрузок. Недостатки по сравнению с PostgreSQL: менее развитые возможности сложных запросов и типов данных, исторически более слабая поддержка полноценных транзакций в отдельных движках. Для приложения с заявками, согласованиями, чатом и архивом ВКР важны надёжные транзакции и богатые возможности SQL, что обеспечивает PostgreSQL.

**Microsoft SQL Server.** Реляционная СУБД корпоративного класса [10]. Достоинства: глубокая интеграция с экосистемой Microsoft, развитые средства администрирования и аналитики. Недостатки для внедрения в вузе: лицензионные ограничения и стоимость при развёртывании на сервере, привязка к экосистеме Windows Server в типовых сценариях. PostgreSQL обеспечивает сопоставимую функциональность без лицензионных затрат и с кроссплатформенным развёртыванием (в том числе в Docker и Linux), что предпочтительно для учебного прототипа.

### 2.3. Выводы по результатам сравнения

Проведённый анализ показывает, что выбранный набор инструментальных средств — **ASP.NET Core 10** (C#) для сервера, **Angular 18** (TypeScript, SCSS) для клиента и **PostgreSQL 16** в качестве СУБД — согласован между собой и соответствует целям проекта:

- Серверная платформа обеспечивает строгую типизацию, встроенную поддержку Web API, аутентификации и авторизации, эффективное использование ресурсов и удобное развёртывание в контейнерах. Аналоги (Java/Spring, Node.js) также пригодны для подобных задач, однако для прототипа с чёткой бизнес-логикой и возможным развёртыванием в вузе ASP.NET Core даёт оптимальное сочетание простоты настройки, производительности и документируемости.
- Клиентский фреймворк Angular обеспечивает единую структуру SPA, типизацию на TypeScript и предсказуемые практики разработки интерфейсов под несколько ролей (студент, преподаватель, заведующий кафедрой, администратор), что упрощает описание в пояснительной записке и дальнейшую поддержку.
- Выбор PostgreSQL обоснован требованиями к надёжному хранению структурированных данных, транзакциям и целостности при отсутствии лицензионных ограничений и хорошей интеграции с EF Core и Docker.

Таким образом, описание выбранных инструментальных средств и их сравнение с аналогами подтверждают обоснованность технологического стека для разработки прототипа сервиса по выбору научного руководителя и темы ВКР в высшем учебном заведении.
